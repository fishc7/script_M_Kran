# Анализ проблемы с расчетом ВИК в таблице сварено_сварщиком

## Проблема

Для чертежа **GCC-NAG-DDD-12460-12-1500-TK-ISO-00052**:
- **Ожидается**: `заявлен_вик = 3` (3 шва со статусом "Заказ отправлен": F39, F130, F158)
- **Фактически в таблице**: `заявлен_вик = 7` (в группе ID=1009)

## Текущая логика расчета ВИК

### 1. При INSERT (строка 736):
```sql
SUM(CASE WHEN TRIM(COALESCE(wc.Результаты_АКТ_ВИК, '')) = 'Заказ отправлен' THEN 1 ELSE 0 END) as заявлен_вик
```

### 2. При UPDATE (строка 1232):
```sql
UPDATE сварено_сварщиком SET ВИК = COALESCE(заявлен_вик, 0)
```

## Проверка данных

### В таблице wl_china для чертежа GCC-NAG-DDD-12460-12-1500-TK-ISO-00052:
- **Всего сваренных швов**: 134
- **Со статусом "Заказ отправлен"**: 3 (F39, F130, F158)
- **Со статусом "годен"**: 131

### В таблице сварено_сварщиком:
- **Группа ID=1009** (клейма: 2Z08, методы: SMAW, типы: У17):
  - `Всего_сваренно_сварщиком = 26`
  - `заявлен_вик = 7` ❌ (должно быть 3)
  - `годен_вик = 19`
  - `ВИК = 7` ❌ (копируется из заявлен_вик)

## Возможные причины проблемы

### 1. Проблема с группировкой
Запрос группирует данные по:
- `N_Линии`
- `_Линия` (вычисляемое поле)
- `Номер_чертежа`
- `клейма_сварщиков` (вычисляемое поле через подзапрос)
- `методы_сварки` (вычисляемое поле через подзапрос)
- `типы_сварных_швов` (вычисляемое поле через подзапрос)

**Проблема**: GROUP BY с вычисляемыми полями может привести к неправильной группировке, если подзапросы возвращают разные результаты для одной и той же записи.

### 2. Проблема с подзапросами для клейм
Подзапрос для `клейма_сварщиков` (строки 654-680):
```sql
SELECT group_concat(part, '/')
FROM (
    SELECT DISTINCT part
    FROM (
        SELECT trim(wc.Клеймо_сварщика_корневой_слой) as part 
        WHERE wc.Клеймо_сварщика_корневой_слой != ''
        UNION ALL
        SELECT trim(wc.Клеймо_сварщика_заполнение_облицовка) as part 
        WHERE wc.Клеймо_сварщика_заполнение_облицовка != ''
    )
    WHERE part IS NOT NULL AND part != ''
)
```

**Проблема**: Этот подзапрос выполняется для каждой строки в GROUP BY, но может возвращать разные результаты в зависимости от контекста группировки.

### 3. Проблема с дублированием записей
Если один шов попадает в группу несколько раз из-за особенностей GROUP BY, то `SUM` будет считать его несколько раз.

## Рекомендации по исправлению

### Вариант 1: Исправить логику подсчета заявлен_вик
Вместо подсчета в основном запросе, можно использовать подзапрос, который точно считает количество уникальных швов:

```sql
(
    SELECT COUNT(DISTINCT wc2.Номер_сварного_шва)
    FROM wl_china wc2
    WHERE wc2.Номер_чертежа = wc.Номер_чертежа
      AND wc2.N_Линии = wc.N_Линии
      AND TRIM(COALESCE(wc2.Результаты_АКТ_ВИК, '')) = 'Заказ отправлен'
      AND (wc2.Клеймо_сварщика_корневой_слой IN (...клейма из группы...)
           OR wc2.Клеймо_сварщика_заполнение_облицовка IN (...клейма из группы...))
) as заявлен_вик
```

### Вариант 2: Исправить GROUP BY
Использовать прямые поля вместо вычисляемых в GROUP BY:

```sql
GROUP BY 
    wc.N_Линии,
    _Линия,
    wc.Номер_чертежа,
    wc.Клеймо_сварщика_корневой_слой,
    wc.Клеймо_сварщика_заполнение_облицовка,
    wc.Метод_сварки_корневой_слой,
    wc.Метод_сварки_заполнение_облицовка,
    wc.Тип_соединения_российский_стандарт
```

А затем объединять клейма и методы через `group_concat` в SELECT.

### Вариант 3: Использовать CTE (Common Table Expression)
Сначала создать временную таблицу с правильной группировкой, а затем считать статистику.
