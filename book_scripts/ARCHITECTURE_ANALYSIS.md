## 2. Анализ компонентов

### 2.1 Веб-приложение (Flask)

**Текущая архитектура:**
```
web/app/
├── app.py (8727 строк) - старый монолитный файл
├── app_new.py (136 строк) - новое модульное приложение
└── modules/
    ├── config.py - конфигурация
    ├── database/db_manager.py - менеджер БД
    ├── routes/ - маршруты (4 модуля)
    ├── api/ - API эндпоинты (2 модуля)
    └── utils/ - утилиты
```

**Проблемы:**
1. **Два файла приложения** - `app.py` и `app_new.py` создают путаницу
2. **Старый app.py (8727 строк)** - монолитный файл, который не используется, но занимает место
3. **Неполная миграция** - не все функции перенесены в модульную структуру
4. **Отсутствие dependency injection** - жесткая связанность компонентов

**Рекомендации:**
- ✅ Переименовать `app_new.py` в `app.py` после полной миграции
- ✅ Переместить старый `app.py` в архив
- ✅ Добавить dependency injection для компонентов
- ✅ Создать фабрику приложений для тестирования

#### Рекомендация 1: Создание базовых классов для ETL

**Что сделать:**
```python
# scripts/core/etl/base_loader.py
class BaseETLLoader(ABC):
    """Базовый класс для всех загрузчиков данных"""
    
    @abstractmethod
    def extract(self, source: str) -> pd.DataFrame:
        """Извлечение данных из источника"""
        pass
    
    @abstractmethod
    def transform(self, df: pd.DataFrame) -> pd.DataFrame:
        """Трансформация данных"""
        pass
    
    @abstractmethod
    def load(self, df: pd.DataFrame) -> bool:
        """Загрузка данных в БД"""
        pass
    
    def run(self, source: str) -> bool:
        """Запуск полного ETL процесса"""
        try:
            df = self.extract(source)
            df = self.transform(df)
            return self.load(df)
        except Exception as e:
            self.logger.error(f"Ошибка ETL: {e}")
            return False
```

**Преимущества:**
- Единый интерфейс для всех загрузчиков
- Легко тестировать
- Переиспользование кода
- Упрощение добавления новых загрузчиков

**Оценка времени:** 16-20 часов
