# Описание проблемы с расчетом ВИК простым языком

## Суть проблемы

**Для чертежа GCC-NAG-DDD-12460-12-1500-TK-ISO-00052:**
- Должно быть: **3 шва** со статусом "Заказ отправлен" (F39, F130, F158)
- В таблице показывается: **7 швов** ❌

**Для титула 12460-12:**
- Должно быть: остаток **13** + не заявлен **1** = **14** всего
- В таблице показывается: **19** + не заявлен **1** = **20** всего ❌
- Лишних: **6 швов** (20 - 14 = 6)

## Как сейчас считается ВИК

### Шаг 1: При создании таблицы (INSERT)
Скрипт группирует данные из таблицы `wl_china` по:
- Номер линии
- Номер чертежа  
- **Клейма сварщиков** (объединяет через `/`)
- **Методы сварки** (объединяет через `/`)
- **Типы сварных швов** (объединяет через `/`)

Для каждой группы считает:
- `заявлен_вик` = сколько швов со статусом "Заказ отправлен"
- `годен_вик` = сколько швов со статусом "годен"
- `не_годен_вик` = сколько швов со статусом "Не годен"
- `не_подан_вик` = сколько швов без статуса

### Шаг 2: При обновлении (UPDATE)
Просто копирует `заявлен_вик` в поле `ВИК`:
```sql
ВИК = заявлен_вик
```

## В чем проблема?

### Проблема №1: GROUP BY с вычисляемыми полями

В запросе используется `GROUP BY` с вычисляемыми полями:
```sql
GROUP BY 
    wc.N_Линии,
    _Линия,
    wc.Номер_чертежа,
    клейма_сварщиков,    -- ← вычисляемое поле (подзапрос)
    методы_сварки,        -- ← вычисляемое поле (подзапрос)
    типы_сварных_швов    -- ← вычисляемое поле (подзапрос)
```

**Проблема**: Подзапросы для клейм, методов и типов выполняются **для каждой строки отдельно**, а не для всей группы. Это может привести к тому, что:
- Одна и та же группа разбивается на несколько записей
- Один и тот же шов попадает в несколько групп
- Подсчет `SUM` работает неправильно

### Проблема №2: Подзапрос для клейм

Подзапрос для `клейма_сварщиков` (строки 654-680):
```sql
SELECT group_concat(part, '/')
FROM (
    SELECT DISTINCT part
    FROM (
        SELECT trim(wc.Клеймо_сварщика_корневой_слой) as part 
        WHERE wc.Клеймо_сварщика_корневой_слой != ''
        UNION ALL
        SELECT trim(wc.Клеймо_сварщика_заполнение_облицовка) as part 
        WHERE wc.Клеймо_сварщика_заполнение_облицовка != ''
    )
    WHERE part IS NOT NULL AND part != ''
)
```

**Проблема**: Этот подзапрос использует `wc.Клеймо_сварщика_корневой_слой` и `wc.Клеймо_сварщика_заполнение_облицовка`, которые относятся к **текущей строке** в группе, а не ко **всей группе**. 

Если в группе есть несколько строк с разными комбинациями клейм, подзапрос может вернуть разные результаты для разных строк, что приведет к неправильной группировке.

## Пример проблемы

**Данные в wl_china:**
- Шов F39: клеймо=2Z08/2Z08, метод=SMAW, тип=У17, статус="Заказ отправлен"
- Шов F130: клеймо=2Z08/2Z08, метод=SMAW, тип=У17, статус="Заказ отправлен"  
- Шов F158: клеймо=2Z08/2Z08, метод=SMAW, тип=У17, статус="Заказ отправлен"
- Шов F03: клеймо=2Z08/2Z08, метод=SMAW, тип=У17, статус="годен"
- ... еще 22 шва со статусом "годен"

**Ожидаемый результат:**
- Одна группа: клейма=2Z08, методы=SMAW, типы=У17
- `заявлен_вик = 3` ✅
- `годен_вик = 23` ✅

**Фактический результат в таблице:**
- Одна группа: клейма=2Z08, методы=SMAW, типы=У17
- `заявлен_вик = 7` ❌ (должно быть 3)
- `годен_вик = 19` ❌ (должно быть 23)

## Почему получается 7 вместо 3?

Возможные причины:
1. **Дублирование записей**: Один и тот же шов попадает в группу несколько раз из-за особенностей GROUP BY
2. **Неправильная группировка**: Группа разбивается на несколько записей из-за разных результатов подзапросов
3. **Ошибка в подсчете SUM**: `SUM` считает один и тот же шов несколько раз

## Решение

### Вариант 1: Исправить GROUP BY
Использовать прямые поля вместо вычисляемых в GROUP BY:

```sql
GROUP BY 
    wc.N_Линии,
    _Линия,
    wc.Номер_чертежа,
    wc.Клеймо_сварщика_корневой_слой,
    wc.Клеймо_сварщика_заполнение_облицовка,
    wc.Метод_сварки_корневой_слой,
    wc.Метод_сварки_заполнение_облицовка,
    wc.Тип_соединения_российский_стандарт
```

А затем объединять клейма и методы через `group_concat` в SELECT (но это нужно делать правильно, чтобы не было дублирования).

### Вариант 2: Использовать CTE (Common Table Expression)
Сначала создать временную таблицу с правильной группировкой по прямым полям, а затем считать статистику и объединять клейма/методы.

### Вариант 3: Исправить подсчет заявлен_вик
Вместо подсчета в основном запросе, использовать подзапрос, который точно считает количество уникальных швов:

```sql
(
    SELECT COUNT(DISTINCT wc2.Номер_сварного_шва)
    FROM wl_china wc2
    WHERE wc2.Номер_чертежа = wc.Номер_чертежа
      AND wc2.N_Линии = wc.N_Линии
      AND TRIM(COALESCE(wc2.Результаты_АКТ_ВИК, '')) = 'Заказ отправлен'
      AND (/* условия по клеймам, методам, типам из группы */)
) as заявлен_вик
```
